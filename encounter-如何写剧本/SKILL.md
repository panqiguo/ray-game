# Encounter Scene Treatment

这个 skill 不直接写 `.enc` 脚本。

它的职责是把用户脑海里零散的点子，整理成一份适合后续实现的"纸面剧本"或"桥段设计稿"。

适用场景：

- 用户只有一个很模糊的点子
- 用户知道"这段戏想讲什么"，但还没想清楚怎么变成可玩的 encounter
- 用户希望先把桥段写成结构化文字，再进入实现

## 最终产物

这个 skill 的输出目标不是代码，而是一份清晰的场景设计稿。它通常包含：

1. 桥段概述
2. 分幕结构
3. 每一幕的目标与阻碍
4. 玩家在这一幕可采用的方法（按地点组织）
5. 关键进度时钟或潜在状态变化
6. 幕与幕之间的流转关系
7. 结束条件

可以把它理解成：

- 不是程序
- 不是配置
- 而是"可以落到程序里的剧本"

## 核心原则

### 1. 先写戏，再写规则

先回答：

- 这一段戏发生了什么
- 玩家此刻想达成什么
- 对手或环境在阻碍什么
- 这一幕为什么紧张

不要一开始就把内容写成 clock 和 action 列表。

### 2. 每一幕只解决一个主要问题

一幕最好只围绕一个清楚的目标展开，比如：

- 潜入豪宅外圈
- 绕过门禁和监控
- 打开保险柜
- 从压制中翻身
- 撬开敌人防守

如果一幕同时在解决太多问题，通常应该拆成两幕。

### 3. 基本单元是地点与行动

游戏里的基本单元只有两种：**地点**和**行动**。

- 地点可以包含子地点，也可以包含行动。
- 行动只存在于某个地点之下。
- **同一地点下的行动可能互相影响**（共享该地点的状态与时钟）。
- **不同地点之间的行动通常不互相影响**（状态隔离）。

写剧本时，一旦两条路线互不干扰，就应该把它们拆成两个独立地点，而不是写在同一个列表里。常见的分叉结构：

```
一幕
 ├─ 地点A：左侧走廊（保安把守）
 │    ├─ 行动：扔石子
 │    └─ 行动：穿越
 └─ 地点B：右侧小径（狗把守）
      ├─ 行动：贴墙慢行
      └─ 行动：投喂毒食
```

在同一地点内，各行动共享进度时钟和局部状态；跨地点的行动则只影响全局状态（如全局警觉度）。

### 4. 每一幕都要说明玩家"怎么解决"

一幕不能只有画面感，还必须给出几类可行方法，并明确它们属于哪个地点。

例如潜入场景里：

- **地点：正门**：撬锁 / 等待时机
- **地点：侧窗**：攀爬 / 借助道具

打斗场景里：

- **地点：狭巷**：防守反击 / 冒险抢武器
- **地点：屋顶**：强攻抢节奏 / 先骗出破绽

### 5. 把变化中的事实记下来

剧本里要写清哪些"事实"会影响后续局面，并说明它属于全局还是某个地点的局部状态。

常见类别：

- 全局时钟（如警觉度，贯穿整个 encounter）
- 局部时钟（如保安巡逻时钟，只在特定地点内计数）
- 布尔状态（如是否拿到道具、敌人是否暴露破绽）
- 枚举状态（如狗的状态：平静 / 激动 / 昏睡）

这里不要求写具体 DSL，但要把这些变化点描述清楚。

### 6. 说明幕与幕之间怎么流转

至少要写清楚：

- 第一幕怎样进入第二幕
- 是否存在分支流转
- 不同地点的路线会不会导向不同幕
- 哪些幕会重新汇合

例如：

- 幕一地点A成功后 → 进入幕二
- 幕一地点B成功后 → 同样进入幕二（两条路汇合）
- 幕一任一路线被发现且逃跑 → 全局警觉度上升，带着代价进入幕二

## 推荐输出格式

默认按下面结构组织：

### 1. 桥段概述

用 2 到 4 句话概括：

- 玩家是谁
- 当前场景是什么
- 玩家想达成什么
- 这段戏的整体气质是什么

### 2. 关键事实

按全局 / 幕专属分组列出会变化的事实。说明每条事实：

- 它代表什么
- 为什么重要
- 满了或见底意味着什么

### 3. 分幕设计

每一幕先写一段有镜头感的场景描述，再按地点展开。

每个地点至少写清：

- 地点名与当前阻碍
- 该地点下的行动列表
- 每个行动的效果（包括成功/部分成功/失败的分叉）
- 该地点内可能变化的局部事实

每一幕最后写转场条件。

建议模板：

```md
### 幕一：外圈潜入

[场景描述，2-3句，有画面感]

**目标**：进入宅邸外圈区域

---

#### 地点A：正门（门卫把守）

**行动：伪装接近**
...效果描述...

**行动：制造干扰**
...效果描述...

局部变化：`门卫警觉度`

---

#### 地点B：侧墙（无人区）

**行动：翻墙**
...效果描述...

---

**转场条件**
- 任意路线完成 → 进入幕二
- 全局警觉度满格 → 行动失败
```

### 4. 流转图

用 ASCII 图或自然语言列出幕与幕、地点与地点之间的连接关系。

### 5. 结束方式

说明这段戏的 2 到 3 种结束方式，对应不同的玩家表现。

## 如何把混乱想法整理成剧本

当用户只给一句模糊描述时，按下面顺序整理：

1. 先确认这段戏的核心目标
2. 拆出 2 到 4 个主要幕
3. 每幕识别有哪些独立的地点分叉
4. 每个地点下补齐行动、效果与局部状态
5. 列出全局事实与各幕专属事实
6. 说明地点与幕之间的流转关系
7. 最后补结束方式

## 进度时钟怎么在剧本里出现

把 clock 当成"会逐步推进的戏剧事实"，区分全局与局部：

- **全局时钟**（如`全局警觉度`）：任何幕、任何地点的失败都可能推进，代表整体风险积累。
- **局部时钟**（如`保安巡逻时钟`）：只在特定地点内计数，代表该地点内的时间窗口或机会窗口。

在剧本里，每条时钟都要说明：它代表什么，它为什么重要，它满了或见底意味着什么。

## 回答风格

输出时优先像"桥段设计文档"而不是"系统说明"。

要有：

- 镜头感
- 戏剧目标
- 行动路线
- 转场逻辑

不要一上来写 AST、编译、render 这些实现术语，除非用户明确要求。
